# Debugging Report

## Overview
This report details the findings from a comprehensive debugging session of the `actual-mcp` codebase. The session included running unit tests, linting, and type checking.

## 1. Unit Test Failures
**Status:** 3 Failures
**File:** `mcp-server/src/core/transport/streamable-http-handler.test.ts`
**Error:** `AssertionError: expected 0 to be greater than 0`

### Analysis
The failures occur in property-based tests using `fast-check`. Specifically:
- `should handle request and connect transport`
- `should handle multiple requests with same handler`
- `should handle cleanup gracefully`

The error `expected 0 to be greater than 0` likely comes from:
```typescript
expect(testHandler.getActiveSessionCount()).toBeGreaterThan(0);
```

This implies that after `testHandler.handleRequest(mockReq, mockRes)` is called, the session count remains 0, meaning no session was established or it was immediately closed.

Since these are property-based tests, they generate random inputs. The counterexamples provided in the logs are:
- `["GET"]`
- `[1]` (numRequests)
- `[0]` (numCleanups)

This suggests that for some inputs (perhaps specific HTTP methods or request structures generated by `fast-check`), the handler fails to establish a session.

### Recommendation
Investigate `StreamableHTTPHandler.handleRequest` to ensure it correctly initializes sessions for all valid inputs. The test might need to be adjusted if certain generated inputs are actually invalid for session creation, or the handler needs to be more robust.

## 2. TypeScript Errors
**Status:** Many Errors
**File:** `mcp-server/src/tools/index.ts` (and others)
**Key Issues:**

1.  **ToolInput Schema Mismatch:**
    ```typescript
    Property 'type' is missing in type 'ToolInput' but required in type '...'
    ```
    The `ToolInput` type used in tool definitions seems to lack the `type` property (likely `type: 'object'`) which is required by the `ToolDefinition` interface or the MCP SDK.

2.  **Argument Type Incompatibility:**
    ```typescript
    Type 'Record<string, unknown>' is missing the following properties...
    ```
    The generic `ToolDefinition` interface expects a handler taking `Record<string, unknown>`, but individual tool handlers are typed with specific arguments (e.g., `{ amount: number; month: string; }`). TypeScript correctly flags this as unsafe because `Record<string, unknown>` does not guarantee the presence of those properties.

### Recommendation
1.  **Fix ToolInput:** Ensure all tool schemas explicitly include `type: 'object'` or update the `ToolInput` type definition to match the expected structure.
2.  **Harmonize Handler Signatures:** Update `ToolDefinition` to allow for specific argument types, or wrap handlers to cast `Record<string, unknown>` to the expected type (using Zod parsing which is already likely happening inside the handler, but the function signature needs to match). A common pattern is:
    ```typescript
    handler: (args: Record<string, unknown>) => {
       const safeArgs = schema.parse(args); // Zod parse
       return actualHandler(safeArgs);
    }
    ```

## 3. Linting Issues
**Status:** 247 Warnings (0 Errors)
**Key Issues:**
- `max-nested-callbacks`: Many functions have deep nesting (4+ levels), making code hard to read and test.
- `max-lines-per-function`: Several functions are too long (>100 lines).
- `complexity`: High cyclomatic complexity in some methods.
- `@typescript-eslint/no-explicit-any`: Usage of `any` type, which defeats type safety.

### Recommendation
- **Refactor Nested Code:** Extract callbacks into named functions.
- **Break Down Large Functions:** Split complex logic into smaller, testable helper functions.
- **Strict Typing:** Replace `any` with specific types or `unknown` where appropriate.

## Conclusion
The codebase is functional but has latent issues that could affect reliability and maintainability. The test failures in the transport layer are the most critical to address as they could indicate issues with connection handling. The TypeScript errors point to potential runtime crashes if data doesn't match expectations, although Zod validation likely mitigates this at runtime. Fixing the type definitions will improve developer experience and catch bugs earlier.
